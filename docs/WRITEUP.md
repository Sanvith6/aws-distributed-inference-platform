# Distributed Inference — Production Hardening & Scaling Writeup

This document addresses the two key questions asked in the assignment:

1. **What would you harden before putting this in production?**
2. **What would you do differently if the model were 100× larger?**

These are not theoretical answers — they reflect direct lessons from building and debugging this system.

---

## Part 1: Production Hardening

### 1.1 Edge Security — Replace Self-Signed SSL with ALB + ACM

The current deployment uses an OpenSSL self-signed certificate generated by Ansible. This is acceptable for internal testing and demonstrations but is unsuitable for any user-facing production traffic because:

- Browsers display certificate warnings, breaking user trust
- Certificate rotation is manual and error-prone
- There is no certificate authority chain for automated validation

**Production path:**
- Deploy an **AWS Application Load Balancer (ALB)** in front of VM1
- Attach an **AWS Certificate Manager (ACM)** certificate (free for public domains, auto-renews)
- The ALB handles TLS termination; VM1 only receives plain HTTP from the ALB

Additionally, attach **AWS Web Application Firewall (WAF)** to the ALB:
- Enable managed rule groups for OWASP Top 10 (SQLi, XSS, command injection)
- Add rate-limiting rules per client IP at the L7 layer
- Configure geo-blocking if the service has regional restrictions

### 1.2 Identity — Eliminate SSH Keys, Use SSM Exclusively

The current deployment relies on an RSA 4096-bit SSH key pair (`iii-key.pem`) stored locally on the administrator's machine. This introduces several operational risks:

- Key theft or accidental exposure (e.g., committed to git)
- No audit trail of who ran which commands when
- Key rotation requires coordinated update across all instances
- SSH port 22 open on the engine VM is an attack surface

**Production path:**
- **Remove port 22 from all security groups entirely**
- Rely exclusively on **AWS Systems Manager Session Manager** for shell access
- SSM routes through HTTPS (port 443) — no inbound firewall changes needed
- Every session is automatically logged to CloudWatch Logs (full audit trail)
- Access is controlled via IAM — revoke immediately by updating the IAM policy

### 1.3 IAM — Narrow Policy Scope

Current IAM roles use AWS managed policies (`AmazonSSMManagedInstanceCore`, `CloudWatchAgentServerPolicy`). These are broader than strictly necessary.

**Production path:**
- Replace managed policies with **inline IAM policies** scoped to exact resources:
  ```json
  {
    "Action": ["ssm:UpdateInstanceInformation", "ssm:StartSession"],
    "Resource": "arn:aws:ec2:us-east-1:123456789:instance/i-specific-id"
  }
  ```
- Create a dedicated **KMS key** for encrypting SSM session data and EBS volumes
- Enable **IAM Access Analyzer** to continuously surface overly permissive policies

### 1.4 Secrets Management

Currently:
- The HuggingFace token (if used) would be hardcoded in the systemd unit file
- AWS credentials live in environment variables on the admin machine
- The SSL private key is on the filesystem of VM1

**Production path:**
- Store all secrets (HF tokens, API keys, database passwords) in **AWS Secrets Manager**
- EC2 instances retrieve secrets at runtime using IAM role permissions (no hardcoded values)
- Rotate secrets automatically using Secrets Manager's rotation scheduler
- Encrypt EBS root volumes with **AWS KMS** (customer-managed keys)

### 1.5 Observability — Centralized Monitoring

Currently, logs are viewed via `journalctl` on individual VMs. There is no centralized visibility into system health or request performance.

**Production path:**

**Metrics:**
- Deploy the **CloudWatch Unified Agent** on all VMs
- Collect: CPU, memory, disk, Nginx request rates, latency histograms
- Create CloudWatch **dashboards** for real-time operational visibility

**Logs:**
- Ship Nginx access/error logs, iii-engine logs, and worker logs to **CloudWatch Log Groups**
- Set log retention policies (e.g., 30 days for debug, 1 year for audit)

**Alerting:**
- CloudWatch Alarms triggered on:
  - Worker disconnect events (inference requests would fail)
  - API error rate > 5% for 5 minutes → PagerDuty
  - VM memory usage > 85% (risk of OOM kill on inference worker)
  - Inference latency P99 > 10 seconds (model performance degradation)

**Distributed Tracing:**
- Enable the iii-engine's built-in OpenTelemetry exporter
- Send spans to Datadog, Grafana Tempo, or AWS X-Ray
- Trace individual requests end-to-end: Nginx → engine → TS worker → Python worker → model

### 1.6 Compute Hardening

**OS Level:**
- Deploy from **CIS Benchmark Ubuntu 22.04 Level 1** hardened AMIs
- Enable `unattended-upgrades` for automatic daily security patches
- Install `fail2ban` on VM1 to block IPs with repeated SSH failures
- Run **AWS Inspector** for automated CVE scanning of running instances

**Process Level:**
- All systemd units already include `NoNewPrivileges=true`, `ProtectSystem=strict`, etc.
- In production, add `MemoryMax=3G` on the inference worker to hard-limit memory and trigger a clean restart rather than an OOM kill

### 1.7 High Availability — Eliminate Single Points of Failure

Current architecture has three SPOFs:
1. **NAT Instance (VM4)** — private workers lose internet if it fails
2. **Engine (VM1)** — entire service fails if it crashes
3. **Single AZ** — AZ failure affects all VMs

**Production path:**

**NAT:**
- Replace NAT instance with **AWS Managed NAT Gateway** (multi-AZ capable, ~$32/mo but fully managed)
- Or deploy NAT instances in an **Auto Scaling Group** with health-checked route replacement

**Engine:**
- Deploy **2+ iii-engine instances** in separate AZs, behind an ALB
- Workers reconnect to any available engine via connection management (requires iii-engine clustering support)

**Multi-AZ:**
- Provision public and private subnets in **at least 2 Availability Zones** (e.g., us-east-1a and us-east-1b)
- Deploy workers in each AZ behind separate Auto Scaling Groups

### 1.8 Immutable Infrastructure with Packer

Current deployments use Ansible to configure running instances — this can lead to **configuration drift** over time (minor differences between instances accumulate through manual fixes, failed playbook runs, or dependency version changes).

**Production path:**
```
GitHub Actions CI/CD
    ↓
Packer + Ansible: Build → Configure → Snapshot
    ↓
Register AMI (e.g., iii-inference-worker-v1.4.2)
    ↓
Terraform: Update AMI ID variable → Rolling update on Auto Scaling Group
    ↓
New instances launch pre-configured → Old instances drained and terminated
```

This ensures every instance is identical to every other instance of the same version, and rollbacks are a single Terraform variable change.

## Part 2: Scaling to a 100× Larger Model

A 100× larger model means transitioning from a **270M parameter SLM (~540MB GGUF)** to a roughly **27B parameter LLM** (~54GB in FP16, or ~18GB in 4-bit quantization).

This is not a scaling problem — it's an **architecture problem**. CPU-based inference becomes completely infeasible, and the system must be redesigned around GPU-native serving.

### 2.1 GPU Compute

| Model Size | Instance Type | VRAM | Suitable For |
|---|---|---|---|
| 27B (Q4 quantized) | g5.xlarge | 24 GiB (1× A10G) | Single-GPU inference |
| 27B (Q8 quantized) | g5.12xlarge | 96 GiB (4× A10G) | Tensor-parallel inference |
| 27B (FP16) | p4d.24xlarge | 320 GiB (8× A100) | High-throughput serving |

For a cost-efficient production deployment serving moderate traffic:
- Start with **g5.xlarge** ($1.00-$2.00/hr) with 4-bit quantization
- Scale horizontally with multiple g5.xlarge instances rather than one large instance

### 2.2 Serving Backend — Replace Custom Python Script

The current Python inference worker is a direct llama.cpp wrapper. For a 27B model, this is replaced with a production-grade serving backend:

**vLLM** (recommended):
```bash
pip install vllm
vllm serve meta-llama/Llama-3-27b --tensor-parallel-size 4
```

Key features that matter at scale:
- **PagedAttention** — prevents KV-cache memory fragmentation, allowing 3-5× more concurrent requests
- **Continuous batching** — processes tokens from multiple requests simultaneously (not waiting for full batches)
- **OpenAI-compatible API** — drop-in replacement for the current Python worker

**Text Generation Inference (TGI)** is an alternative with similar capabilities.

### 2.3 Queue-Based Decoupling

With a 27B model, a single inference call takes **10-60 seconds**. Synchronous HTTP with a 120-second timeout is inadequate for burst traffic. The architecture shifts to async:

```
Caller-worker (API tier)
    ↓ (enqueue request + return job_id immediately)
Amazon SQS queue
    ↓ (workers poll for jobs)
vLLM worker pool (GPU instances)
    ↓ (write result to S3/DynamoDB)
Client polls GET /v1/jobs/{job_id}/result
```

This decouples client response time from inference time, absorbs traffic spikes, and enables horizontal scaling of the GPU tier independently of the API tier.

### 2.4 Kubernetes on EKS — Container Orchestration

For managing a fleet of GPU inference workers, bare EC2 instances become operationally difficult. Kubernetes (via EKS) provides:

**Auto-scaling:**
```yaml
# Karpenter NodePool for GPU nodes
apiVersion: karpenter.sh/v1alpha5
kind: NodePool
spec:
  template:
    spec:
      requirements:
        - key: node.kubernetes.io/instance-type
          operator: In
          values: ["g5.xlarge", "g5.2xlarge"]
```

**Pod-level resource guarantees:**
```yaml
resources:
  requests:
    nvidia.com/gpu: "1"
  limits:
    nvidia.com/gpu: "1"
```

**Node pre-warming:**
- Cache model weights as an EBS snapshot
- New GPU pods mount the snapshot — model loads in seconds instead of downloading from S3

### 2.5 Model Weight Storage Strategy

A 27B model in FP16 is ~54GB. Storage and loading strategy is critical:

| Approach | Load Time | Cost | Notes |
|---|---|---|---|
| Download from HuggingFace on start | 10-30 min | Bandwidth cost | Completely infeasible for production |
| S3 with VPC endpoint | 5-15 min | $0.023/GB/mo storage | Acceptable for infrequent cold starts |
| EBS snapshot pre-cached | 30-60 sec | EBS pricing | Best for warm-pool instances |
| EFS shared model volume | 2-5 min | $0.30/GB/mo | Good for multi-node tensor parallelism |

**Recommended:** Pre-bake model weights into an EBS snapshot using a Packer build pipeline. New instances restore from the snapshot and start serving in under 2 minutes.

## Summary

| Dimension | Demo System (Current) | Production System |
|---|---|---|
| **SSL** | Self-signed | ACM + ALB |
| **SSH Access** | SSH key (22 open) | SSM only (22 closed) |
| **NAT** | t3.nano NAT instance | Managed NAT Gateway |
| **Scaling** | Static EC2 instances | Auto Scaling Groups + EKS |
| **Monitoring** | journalctl | CloudWatch + tracing |
| **Secrets** | Environment variables | AWS Secrets Manager |
| **Model size** | 270M parameters (CPU) | 27B parameters (GPU/vLLM) |
| **Request handling** | Synchronous HTTP | Async SQS + polling |
| **HA** | Single AZ | Multi-AZ, multi-engine |
| **Cost (monthly)** | ~$5-15 | $1,000-$10,000+ depending on traffic |
