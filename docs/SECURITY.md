# Security Design Documentation

This document describes the security architecture of the distributed inference platform: how network isolation is enforced, how access is controlled, and what hardening measures are in place. It also outlines the production recommendations for transitioning this deployment into a fully enterprise-hardened system.

## 1. Security Philosophy

This deployment is designed around **defense-in-depth**: multiple overlapping security controls such that compromising one layer does not compromise the entire system.

The core security properties guaranteed by the architecture:

| Property | How It's Enforced |
|---|---|
| **Workers unreachable from internet** | Private subnet + no public IPs + SG rules |
| **Single public entry point** | Only VM1 has a public Elastic IP |
| **Encrypted client traffic** | Nginx TLS 1.2/1.3 termination at edge |
| **Engine not directly accessible** | Binds to `127.0.0.1` loopback |
| **No lateral movement via SGs** | Workers only accept SSH from VM1's specific private IP |
| **Privileged escalation blocked** | systemd `NoNewPrivileges=true` on all services |
| **Secrets not in codebase** | SSH key generated by Terraform, never committed |

## 2. Network Isolation — Private Subnet Architecture

### The Core Principle

The fundamental security guarantee of this architecture is **network-level isolation**: the AI inference workers (VM2, VM3) are deployed in a **private subnet** with no route to the internet as an ingress destination and no public IP addresses.

```
PUBLIC SUBNET (10.0.1.0/24)
  VM1 — Engine/Nginx     ← has Elastic IP, Internet-reachable
  VM4 — NAT Instance     ← has Elastic IP, Internet-reachable

PRIVATE SUBNET (10.0.2.0/24)
  VM2 — inference-worker ← NO public IP, NOT internet-reachable
  VM3 — caller-worker    ← NO public IP, NOT internet-reachable
```

Any attempt to connect to VM2 or VM3 from the public internet will fail at the **routing layer** — there is simply no route for inbound traffic from the internet to reach the private subnet. This is not a firewall rule that could be misconfigured; it's a topological property of the VPC route tables.

### VPC Route Table Analysis

**Public Route Table** (`iii-public-rt`):
- `10.0.0.0/16` → local (internal VPC traffic)
- `0.0.0.0/0` → Internet Gateway (internet-bound traffic exits here)

**Private Route Table** (`iii-private-rt`):
- `10.0.0.0/16` → local (internal VPC traffic)
- `0.0.0.0/0` → NAT Instance ENI (outbound-only; internet cannot initiate connections back in)

The private route table has **no inbound route from the internet**. The NAT instance allows *outbound* connections (workers downloading packages) but TCP connection state ensures inbound-initiated connections from the internet are dropped.

## 3. Security Groups — Firewall Rules

Security groups in AWS are **stateful firewalls** — return traffic for established connections is automatically allowed. The following rules define the exact security posture of each VM.

### VM1 — Engine Gateway Security Group (`iii-engine-sg`)

| Direction | Protocol | Port | Source | Purpose |
|---|---|---|---|---|
| Ingress | TCP | 22 | `var.my_ip` (admin IP) | SSH admin access |
| Ingress | TCP | 80 | `0.0.0.0/0` | HTTP (redirects to HTTPS) |
| Ingress | TCP | 443 | `0.0.0.0/0` | HTTPS API |
| Ingress | TCP | 49134 | `10.0.2.0/24` | WebSocket RPC from workers |
| Egress | All | All | `0.0.0.0/0` | Outbound connectivity |

**Key security decisions:**
- Port 49134 (WebSocket RPC) is restricted to the **private subnet CIDR** only — outside attackers cannot register fake workers even if they could reach VM1
- Port 22 is restricted to a specific admin IP (`/32`) — not open to the world

### VM2 — Inference Worker Security Group (`iii-inference-sg`)

| Direction | Protocol | Port | Source | Purpose |
|---|---|---|---|---|
| Ingress | TCP | 22 | `VM1 private IP /32` | SSH via bastion only |
| Egress | All | All | `0.0.0.0/0` | Outbound (via NAT) |

**Key security decision:** SSH access is restricted to VM1's **specific private IP address** — not even the entire private subnet (`10.0.2.0/24`). Only the bastion can SSH in.

### VM3 — Caller Worker Security Group (`iii-caller-sg`)

| Direction | Protocol | Port | Source | Purpose |
|---|---|---|---|---|
| Ingress | TCP | 22 | `VM1 private IP /32` | SSH via bastion only |
| Egress | All | All | `0.0.0.0/0` | Outbound (via NAT) |

Identical posture to VM2.

### VM4 — NAT Instance Security Group (`iii-nat-sg`)

| Direction | Protocol | Port | Source | Purpose |
|---|---|---|---|---|
| Ingress | All | All | `10.0.2.0/24` | Traffic from private subnet |
| Ingress | TCP | 22 | `var.my_ip` | SSH admin access |
| Egress | All | All | `0.0.0.0/0` | All outbound traffic |

## 4. NAT Instance Security

The NAT instance presents a unique security challenge: it must accept *all traffic from the private subnet* in order to forward it to the internet. However, several mitigations limit the blast radius:

1. **Source IP Restricted:** The NAT SG ingress allows only `10.0.2.0/24` — only private subnet VMs can send traffic through it. Public internet cannot send arbitrary traffic to the NAT instance's private IP.

2. **No HTTP/HTTPS exposed publicly:** Port 80 and 443 are NOT open on the NAT SG — the NAT instance is not a web server.

3. **iptables MASQUERADE is directional:** The `POSTROUTING MASQUERADE` rule only applies to packets exiting via the default interface. Inbound packets from the internet are not manipulated.

4. **SSH restricted to admin IP:** Port 22 on the NAT SG allows only the admin IP — not the private subnet and not `0.0.0.0/0`.

## 5. TLS and Transport Security

### Certificate Configuration

The Nginx reverse proxy terminates TLS using a **self-signed RSA certificate** generated during Ansible deployment:

```bash
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/nginx/ssl/iii-api.key \
  -out /etc/nginx/ssl/iii-api.crt \
  -subj "/CN=iii-api-gateway/O=iii/C=US"
```

### TLS Protocol and Cipher Configuration

Only modern, secure protocols and cipher suites are enabled:

```nginx
ssl_protocols             TLSv1.2 TLSv1.3;
ssl_prefer_server_ciphers on;
ssl_ciphers               'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:
                            ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:
                            DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384';
ssl_session_cache         shared:SSL:10m;
ssl_session_timeout       1d;
ssl_session_tickets       off;
```

- **TLS 1.0 and 1.1 are disabled** — protection against POODLE and BEAST
- **Session tickets are disabled** — prevents forward secrecy weakening from ticket key rotation
- **Only ECDHE/DHE cipher suites** — ensures Perfect Forward Secrecy (PFS)

### Internal Traffic

Traffic between Nginx and iii-engine (`127.0.0.1:3111`) is **unencrypted plaintext** — this is acceptable because it never leaves the host machine (loopback interface). Encrypting loopback traffic provides no security benefit.

Traffic between the engine and workers over the private subnet (`10.0.1.x ↔ 10.0.2.x`) is also unencrypted WebSocket (`ws://` not `wss://`). This traffic never traverses the public internet — it stays entirely within the VPC. In a production hardening step, `wss://` with private CA certificates would be added.

## 6. IAM — Identity and Least Privilege

### Role Separation

Two separate IAM roles are defined — the engine and workers have different permissions needs:

**`iii-engine-role`** (for VM1):
- `AmazonSSMManagedInstanceCore` — allows SSM Session Manager access (SSH-free admin)
- `CloudWatchAgentServerPolicy` — allows sending metrics and logs to CloudWatch

**`iii-worker-role`** (for VM2, VM3):
- `AmazonSSMManagedInstanceCore` — SSM access only

Workers do not have CloudWatch permissions — they have the minimum required to function.

### Instance Profile Binding

IAM roles are attached via Instance Profiles:

```hcl
resource "aws_iam_instance_profile" "engine" {
  name = "iii-engine-instance-profile"
  role = aws_iam_role.engine.name
}
```

Terraform ensures the correct profile is attached to the correct instance at creation time — no manual console steps.

### What These Roles Cannot Do

The IAM roles **cannot**:
- Create or delete EC2 instances
- Modify VPC or security group rules
- Access S3 buckets (unless explicitly added)
- Assume other IAM roles
- Access other AWS services not explicitly listed

## 7. Bastion Jump Host Access

Since worker VMs (VM2, VM3) have no public IPs, all administrative SSH access routes through VM1 as a **bastion host**:

```
Admin Laptop ──SSH──► VM1 (public, bastion) ──SSH──► VM2/VM3 (private)
```

### SSH Key Management

- SSH key pair is generated by Terraform's `tls_private_key` resource (RSA 4096-bit)
- The private key is written to `terraform/iii-key.pem` (excluded from git via `.gitignore`)
- The public key is uploaded as an AWS Key Pair to the account
- The same key is reused across all VMs — simplifying the bastion hop

### ProxyCommand Pattern

```bash
ssh -i terraform/iii-key.pem \
  -o ProxyCommand="ssh -i terraform/iii-key.pem -W %h:%p ubuntu@<BASTION_IP>" \
  ubuntu@<PRIVATE_VM_IP>
```

This uses SSH's native `ProxyCommand` / `ProxyJump` feature — no third-party tools required.

### SSM Alternative (Zero SSH)

All VMs have `AmazonSSMManagedInstanceCore` attached. AWS Systems Manager Session Manager allows shell access **without SSH keys, without open port 22, and without network routing changes**:

```bash
aws ssm start-session --target i-0abcdef1234567890
```

In a production environment, **port 22 would be removed entirely from all security groups** and SSM would be the only administrative access path.

## 8. WebSocket Isolation Model

The WebSocket RPC architecture provides an inherent security property: **workers only connect outbound**.

### Traditional Model (Vulnerable)
```
Internet → (firewall) → Worker HTTP listener (port 8000)
                           ↑ requires open inbound port
                           ↑ worker is an attack surface
```

### iii-Engine Model (Secure)
```
Worker → (outbound TCP) → iii-engine :49134
        ↑ no inbound ports needed
        ↑ worker accepts no external connections
        ↑ SG can block all inbound to workers
```

This means:
- Workers are **not servers** from a networking perspective — they're clients that happen to process RPC calls
- Compromising the iii-engine does not give direct shell access to workers (no inbound paths)
- Workers can be behind any NAT without configuration changes

### WebSocket Port Restriction

The iii-engine's WebSocket port (49134) is restricted to `10.0.2.0/24` in the security group. This means:
- Only private subnet VMs can register as workers
- External attackers cannot register malicious workers even if they breach VM1's HTTPS layer

## 9. Application-Layer Security (Nginx)

### Rate Limiting

```nginx
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

location /v1/ {
    limit_req zone=api_limit burst=20 nodelay;
}
```

- Limits each client IP to **10 requests/second** sustained, with a burst allowance of 20
- Protects against naïve DoS and API abuse
- In production, this would be supplemented with AWS WAF

### Security Response Headers

```nginx
add_header X-Frame-Options       "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-XSS-Protection      "1; mode=block" always;
add_header Content-Security-Policy "default-src 'none'; frame-ancestors 'none';" always;
```

- **X-Frame-Options: DENY** — prevents clickjacking
- **X-Content-Type-Options: nosniff** — prevents MIME-type sniffing
- **X-XSS-Protection** — enables browser XSS filter (deprecated but harmless)
- **Content-Security-Policy** — restricts resource loading to nothing (API-only, no browser UI)

### HTTP → HTTPS Redirect

All HTTP traffic (port 80) is permanently redirected to HTTPS:
```nginx
server {
    listen 80;
    return 301 https://$host$request_uri;
}
```

## 10. systemd Service Hardening

All three application services (iii-engine, inference-worker, caller-worker) use systemd security options to limit the blast radius of a compromised process:

```ini
[Service]
User=iii                        # Runs as non-root 'iii' user
Group=iii
NoNewPrivileges=true            # Cannot gain additional privileges (no setuid)
ProtectSystem=strict            # Filesystem mounted read-only except ReadWritePaths
ProtectHome=true                # Cannot access /home
PrivateTmp=true                 # Private /tmp namespace (isolated from other services)
ReadWritePaths=/opt/iii         # Only this path is writable
ProtectControlGroups=true       # Cannot modify cgroup settings
ProtectKernelModules=true       # Cannot load/unload kernel modules
ProtectKernelTunables=true      # Cannot modify kernel parameters
RestrictRealtime=true           # Cannot use real-time scheduling
```

This means a compromised worker process **cannot**:
- Access root or gain privileges
- Read files in `/home`, `/root`, or `/etc` (beyond what's needed)
- Write to system directories
- Affect other running services via shared `/tmp`
- Load malicious kernel modules

## 11. Secrets Management

### Current Approach

| Secret | Storage Method |
|---|---|
| SSH private key (`iii-key.pem`) | Local filesystem, `.gitignore` excluded |
| AWS credentials | Environment variables (never committed) |
| SSL private key | On VM1 at `/etc/nginx/ssl/iii-api.key` (root-owned, mode 600) |
| HuggingFace token (if needed) | Environment variable in systemd unit file |

### What Is Not In Git

The `.gitignore` explicitly excludes:
```
*.pem          # SSH private keys
.terraform/    # Terraform provider cache
terraform.tfstate*  # Infrastructure state (may contain sensitive values)
.env           # Environment variable files
venv/          # Python virtual environments
```

## 12. Production Hardening Recommendations

The following improvements are recommended before putting this system into production:

### Network
- Replace self-signed SSL with **AWS Certificate Manager (ACM)** certificates on an **Application Load Balancer (ALB)**
- Attach **AWS WAF** to the ALB with managed rules for OWASP Top 10 protection
- Add **VPC Flow Logs** to monitor and audit all network traffic
- Upgrade private subnet workers to use **wss://** (WebSocket over TLS) with private CA certificates
- Consider adding a **VPC Endpoint for S3** to keep model downloads off the public internet

### Access Control
- **Remove port 22 from all security groups** and rely exclusively on SSM Session Manager
- Enable **SSM Session Manager logging** to CloudWatch for a full audit trail of all admin shell commands
- Tighten IAM policies from managed policies to **inline, resource-scoped policies** (e.g., restrict S3 to specific bucket ARN)
- Enable **IAM Access Analyzer** to detect overly permissive policies

### Secrets
- Migrate credentials and API keys to **AWS Secrets Manager** with IAM-role-based secret access
- Rotate the SSH key pair on a schedule using AWS Key Management Service (KMS)
- Enable **EBS encryption at rest** for all volumes

### Monitoring
- Deploy **CloudWatch Agent** on all VMs for CPU, memory, and disk metrics
- Set CloudWatch Alarms on:
  - `inference-worker` service restarts (> 3 in 5 min → PagerDuty alert)
  - API error rate (5xx > 5% → alert)
  - Memory usage > 85% (→ scale up warning)
- Enable **AWS GuardDuty** for threat detection and anomaly alerts

### Compute Hardening
- Use hardened **CIS Benchmark Ubuntu 22.04** AMIs as the base image
- Enable **automatic security patching** via `unattended-upgrades`
- Deploy `fail2ban` on VM1 to block IPs with repeated SSH failures
- Enable **AWS Inspector** for automated vulnerability scanning of EC2 instances

## 13. Security Checklist

Use this checklist when validating the security posture of a deployment:

### Network Isolation
- [ ] VM2 and VM3 have no public IP addresses
- [ ] Private route table routes `0.0.0.0/0` to NAT ENI (not IGW)
- [ ] NAT instance has `source_dest_check = false`
- [ ] `curl https://www.google.com` succeeds from private VMs (outbound works)
- [ ] Attempting to connect to VM2/VM3 from the internet times out

### Security Groups
- [ ] Port 49134 on VM1 restricted to `10.0.2.0/24` only
- [ ] VM2 and VM3 SSH (`22`) restricted to VM1's private IP `/32` only
- [ ] Port 22 on VM1 restricted to admin IP (not `0.0.0.0/0`)
- [ ] No unnecessary inbound ports open on any VM

### TLS
- [ ] Port 80 returns `301 Moved Permanently` to HTTPS
- [ ] `curl -k https://<ENGINE_IP>/health` returns 200
- [ ] `curl https://<ENGINE_IP>/health` (without `-k`) warns about self-signed cert (expected)
- [ ] `ssl_protocols` includes only TLSv1.2 and TLSv1.3

### Application
- [ ] iii-engine HTTP (port 3111) not accessible from outside VM1
- [ ] `/health` endpoint returns JSON without revealing sensitive info
- [ ] Security headers present in Nginx responses
- [ ] Rate limiting enforced (repeated rapid requests return 429)

### IAM
- [ ] Roles are attached to correct instances via instance profiles
- [ ] Workers do not have engine-level permissions (check role attachments)
- [ ] `aws sts get-caller-identity` from instances uses the correct role ARN

### Secrets
- [ ] `*.pem` files are in `.gitignore` and not committed
- [ ] `terraform.tfstate` is not committed to the repository
- [ ] No credentials are hardcoded in any source file
